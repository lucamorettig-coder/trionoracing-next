Note di design · scope MVP EVO-020

  1. Layout single-column stack: 4 section-card verticali (Anagrafica · Figli · Iscrizioni · Titoli). Scelta vs. 2-colonne perché Titoli/Iscrizioni necessitano larghezza piena per tabelle dense. Coerente con dettaglio bambino EVO-016.
  2. Anagrafica grid 3 colonne: pattern label small uppercase + valore. CLERK_USER_ID in monospace small per debugging dev support.
  3. Figli card orizzontale: avatar colorate per genere/identità (varianti sky/pink), badge categoria FCI + stato iscrizione 2026. CTA "Apri scheda figlio" inline. Riuso FiglioCard da portale EVO-014 in versione admin compatta se compatibile, altrimenti nuovo.
  4. Iscrizioni / Titoli mini-table: stesso pattern DataTable<T> ma in versione compatta (font-size 12.5px, padding ridotto). MethodTag SumUp gradient cyan→blue come EVO-018.
  5. CTA "Cambia ruolo" primario in alto (azione critica visibile sempre). "Apri come admin" visibile ma disabled con tooltip "feature post-MVP" — placeholder per impersonation futura (EVO-008).
  6. Server Action transazionale: il flow Cambia ruolo è una transazione Airtable+Clerk (vedi DS-NOTES §8). Modal richiede 2 step (form + AlertDialog conferma finale) per ridurre errori, soprattutto promozione/declassamento ADMIN.
  7. Gestione errore rollback: se Clerk fallisce, Airtable viene ripristinato e mostriamo toast flag-red con istruzione (vedi DS-NOTES §8 e sezione modal sotto).
  8. Etichette ruolo UI: badge "Maestro" (non "Istruttore"), in coerenza con tabella lista (DS-NOTES §3).
