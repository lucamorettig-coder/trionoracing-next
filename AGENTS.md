## Portale — Pattern e convenzioni (F3)

### Client Airtable portale
- File: `src/lib/airtable-portale.ts` (separato da `airtable-209.ts`)
- Pattern: fetch REST API Airtable, nessun SDK. Env: `AIRTABLE_BASE_ID` + `AIRTABLE_TOKEN`.
- Tabella principale: `TABELLA_GENITORI` (field names in MAIUSCOLO_UNDERSCORE).
- Usa sempre `stripReadOnlyFields()` prima di ogni write Airtable (evita 422 su campi formula/lookup).
- Espandi questo file in future EVO aggiungendo funzioni per nuove tabelle.

### Webhook Clerk
- File: `src/app/api/clerk/webhook/route.ts`
- Runtime: nodejs (non Edge) — richiesto da svix.
- Pattern verifica: `new Webhook(secret).verify(rawBody, headers)`
- `user.created`: crea/aggiorna record TABELLA_GENITORI + setta `publicMetadata.role` via clerkClient.

### Middleware (proxy.ts)
- Protegge `/portale/*` eccetto `/portale/login` e `/portale/registrati`.
- RUOLO letto da `auth.sessionClaims?.role` (configurato nel JWT template Clerk Dashboard).
- Fallback se role undefined: tratta come 'GENITORE'.
- Guard admin: `/portale/admin/*` → solo ADMIN. Guard istruttore: `/portale/lezioni/*`, `/portale/gare-assegnate/*` → ISTRUTTORE + ADMIN.

### NavBar portale
- Componente: `src/components/portale/PortaleNavBar.tsx` (Server Component)
- NavLinks: `src/components/portale/NavLinks.tsx` (Client Component — usePathname per active state)
- MobileMenu: `src/components/portale/MobileMenu.tsx` (Client Component — hamburger + dropdown fixed)
- Ruolo letto da `auth()` server-side — mai dal client.
- Link mostrati variano per GENITORE / ISTRUTTORE / ADMIN.

### Struttura route portale
- `src/app/portale/(portal)/` — route group con layout NavBar (protette da auth)
- `src/app/portale/login/` e `src/app/portale/registrati/` — fuori dal gruppo, nessuna NavBar
- `src/app/portale/dashboard/page.tsx` — permanentRedirect a `/portale`
- Il route group `(portal)` non crea segmento URL — `/portale/(portal)/page.tsx` → URL `/portale`

### Deploy
- Vercel collegato a GitHub (lucamorettig-coder/trionoracing-next).
- Branch principale: main. Pattern: branch dedicato → PR → merge → deploy automatico.

### Pattern appresi in EVO-004 — hotfix 2026-05-24

- **`return_url` SumUp = notification URL per-checkout, non redirect del browser**: il campo `return_url` del payload `POST /v0.1/checkouts` (SumUp API v0.1) è l'URL che SumUp chiama in async quando il checkout cambia stato. Va passato ad **ogni** creazione di checkout, e punta al webhook Make.com (env `MAKE_SUMUP_RETURN_URL`) che fa GET su SumUp e aggiorna Airtable (`PAGATO=true`) in modo indipendente dal browser. È il meccanismo di fallback per il caso "utente paga ma chiude il tab prima del `/verify`" — senza, il pagamento resta `PENDING` su Airtable finché qualcuno non riapre la pagina. Pattern ereditato dal legacy Astro (`area-riservata-triono/src/pages/api/pagamenti/sumup/checkout.ts`).
- **Quando si tocca il payload SumUp, includere sempre `return_url`**: già successo — omesso in EVO-004 (PR #13, commit `b37c8e2`) con regressione del fallback Make.com. Spread condizionale `...(returnUrl ? { return_url: returnUrl } : {})` + `console.warn` se la env è assente: degrado, non errore bloccante.
- **Il merchant-level webhook in dashboard SumUp NON è usato**: la notifica è sempre per-checkout via `return_url` nel payload. Non configurare webhook globali in dashboard SumUp pensando che coprano il caso — non lo fanno.
- **Recovery 409 `DUPLICATED_CHECKOUT` non rifà il payload**: il ramo che riusa un checkout esistente (via `GET /v0.1/checkouts?checkout_reference=...`) non deve aggiungere `return_url` — il checkout già esiste lato SumUp con i parametri originali. Modifiche al payload vanno applicate solo al ramo "create".

### Pattern appresi in EVO-015 (2026-05-24)

- **Campo schema dati come "label primaria umana" quando campi singoli sono inaffidabili**: introdurre un campo dedicato `DESCRIZIONE` (singleLineText) popolato dal punto-creazione (`createIscrizione` per la prima rata, scenario Make.com per rate mensili) invece di costruire la label runtime concatenando campi opzionali. Pattern applicato a `TITOLI_PAGAMENTO` per risolvere "undefinedª rata" da `NUMERO_RATA` non popolato.
- **Helper puro centralizzato per label complesse**: quando ≥3 consumer concatenano campi per costruire una label semantica (es. "Rata 1 · gennaio 2026"), estrarre `titoloLabel(t) → { primary, secondary, secondaryVariant }` in `src/lib/portale-utils.ts` e refactorare i consumer. Elimina mappe locali duplicate e template inline fragili. Pattern applicato a 5 consumer (TabPagamenti, PagamentiLista, StepSommario, DashboardGenitore, checkout).
- **Server Component come "etichetta semantica"**: `<TitoloLabel titolo={t} />` in `src/components/portale/pagamenti/` rende l'output di `titoloLabel()` con styling consistente. Pattern preferibile al template inline quando la label è composta da primary + secondary con variant (es. tipografia diversa, badge accanto).
- **Fallback per tipi sconosciuti in singleSelect**: `TITOLI_PAGAMENTO.TIPO_TITOLO` non è solo `prima_rata`/`rata` — Airtable consente di aggiungere causali manuali (abbigliamento, una tantum). Helper deve gestire default → "Pagamento" invece di assumere enum stretto.
- **Make.com Array Aggregator per evitare duplicati su batch**: quando uno scenario Search Records ritorna N bundle e il flusso a valle creerebbe N record duplicati (es. "genera titolo rata mensile" trova N iscrizioni attive, ma vuoi 1 titolo per iscrizione), inserire un modulo Array Aggregator dopo il Search per collassare i bundle e iterare 1 volta sola.
- **Make.com mapping NUMERO_RATA via aggregator length**: `NUMERO_RATA = length(aggregator.array) + 1` calcola la rata corrente contando quelle pregresse della stessa iscrizione, robusto a re-run dello scenario.
- **Make.com filtro `OR(TIPO_TITOLO=prima_rata, =rata)`**: quando conti rate per determinare il numero progressivo, filtra esplicitamente per i tipi che rappresentano rate vere — escludi causali manuali (abbigliamento, una tantum) che inquinerebbero il conteggio.
- **Schema Airtable additivo è low-risk**: aggiungere campi opzionali a una tabella in produzione è retro-compatibile (record esistenti restano validi con valore vuoto). UI deve sempre avere fallback (`?? 'Pagamento'`) per il periodo di transizione, evitando backfill bloccante.

### Pattern appresi in EVO-006 (2026-05-24)

- **Bug ARRAYJOIN su linked records in filterByFormula — workaround via inverse relationship**: `ARRAYJOIN({CAMPO_LINK},",")` su un campo `multipleRecordLinks` restituisce il **primary field** dei record linkati (formula o singleLineText), NON il loro record ID. Quindi `FIND("recXXX", ARRAYJOIN({CAMPO_LINK},","))` non matcha mai. Manifestazione classica: ricerca cross-tabella tramite linked record fallisce silenziosamente (ritorna sempre 0 record). **Soluzione preferita**: usare l'**inverse relationship** sull'entità che già contiene gli IDs giusti come array. Esempi: invece di cercare lezioni con `FIND("maestroId", ARRAYJOIN({MAESTRO_COMPILATORE},...))`, fetch `TABELLA_MAESTRI/maestroId` e leggi `LEZIONI_COME_COMPILATORE` + `LEZIONI_COME_MAESTRO`. **Soluzione alternativa** (quando l'inverse non esiste): aggiungere un campo lookup `XXX_RECORD_ID_LOOKUP` come fatto in `TABELLA_BAMBINI.GENITORE_RECORD_ID_LOOKUP` e `TABELLA_ISCRIZIONI.GENITORE_RECORD_ID_LOOKUP`. **Soluzione tampone** (quando si ha già il record di partenza in mano): leggi il primary field via record-by-id, deriva l'email/identificativo univoco e fai cross-lookup con quello (es. `getMaestroByGenitoreId` ridisegnata via `EMAIL_GENITORE` match dopo fetch del genitore).

Sintomo: "il bottone non reagisce", nessun errore in console. Pattern: **separare i tipi in `actions-types.ts`** (no direttiva) e importarli via `import type` dentro `actions.ts`. - **AlertDialog DS-alignment con buttonVariants wrappati su Action/Cancel**: di default Radix `AlertDialogPrimitive.Action` e `AlertDialogPrimitive.Cancel` sono esposti come componenti raw senza styling. - **setState durante render con state-snapshot bailout (alternativa a `key={prop}` remount)**: la regola lint React 19 `react-hooks/set-state-in-effect` rifiuta `setState` sincrono dentro `useEffect`. Quando serve **re-inizializzare state derivato da prop che cambia** (es. modal che si apre con un genitore diverso → re-init `nuovoRuolo` da `genitore.fields.RUOLO`), il pattern `key={prop}` remount funziona ma rompe close animations. Alternativa: pattern React ufficiale "Storing information from previous renders" (react.dev) — uno state snapshot della prop + setState durante render con bailout check:
  ```ts
  const [prevKey, setPrevKey] = React.useState<string | null>(null);
  const currentKey = open && genitore ? genitore.id : null;
  if (currentKey !== prevKey) {
    setPrevKey(currentKey);
    if (currentKey && genitore) {
      const targetRuolo = genitore.fields.RUOLO ?? "GENITORE";
      if (nuovoRuolo !== targetRuolo) setNuovoRuolo(targetRuolo);
    }
  }
  ```
  React garantisce che setState sincrono durante render con bailout (gating sull'uguaglianza) non causa loop. **NON usare `useRef` per il prev value** — il lint rule `react-hooks/refs` blocca ref.current access durante render. Sempre `useState` per il snapshot.
- **DEV/PROD schema sync è un debito tecnico ricorrente**: pattern AGENTS.md "macro-task 0 sync schema" già documentato post EVO-016 + EVO-019, ma in EVO-006 il campo `NOTE_INTERNE` su `TABELLA_LEZIONI` era stato applicato solo PROD. EVO-020 ha richiesto creazione lezione su DEV e ha trovato il disallineamento al runtime (`UNKNOWN_FIELD_NAME: NOTE_INTERNE` 422). Pattern consolidato: **prima di ogni mutation cross-tabella in una nuova area, verificare `get_table_schema` su entrambe le basi** anche se la tabella sembra "stabile". Costo trascurabile, evita iterazioni post-smoke per allineamento schema legacy.
